过年假期最后一天，我终于把一个拖了很久的小工程补完，做完的那一刻非常满足。

## 背景：为什么要做这件事

我听小宇宙的播客通常很长，听到后面容易忘记前面；更麻烦的是，节目里经常提到很多 reference，我听的时候来不及逐个记下来。

我想要一个东西：**听完之后能快速回顾要点、定位引用来源，把"播客内容"变成一个可检索的知识库。**

## 我做的系统是什么：一条流水线 + 批处理能力

核心流程是四步：

1. 从小宇宙下载指定节目
2. 本地转写（transcribe）
3. 用大模型做总结（summarize），抽出关键观点与 reference
4. 把结果同步到一个线上空间，方便随时回看

后续我加了批量处理：可以把一组节目列表一次性丢进去，系统会持续监控、自动下载，也能回溯下载很久之前的多期内容，然后批量转写与总结。

## 我踩过的坑：把 AI 串进多步骤流程后，可控性会下降

第一版我把它做成一个 OpenClaw 的 skill：里面有脚本负责下载、转写；总结则用 OpenClaw 内置的 summarize 能力。

从我给出指令,到 OpenClaw 完成, 一共只花了6分钟. 这效率让我有点惊讶, 而且主要功能测试下来是可用的. 但问题出在"步骤一多，变量就多"。总结质量波动明显：有时结构完整，有时遗漏 reference，有时输出格式不稳定。这种不稳定, 导致这套系统最终没有达到可用的阈值。

### 学到的第一点：追求可控性时，固化 prompt + API 编排更稳定

当目标是稳定产出、可重复执行时，把关键 prompt 固化在代码里，通过 API 编排流程，通常比把所有步骤塞进交互式 skill 更容易得到一致结果。

交互式模式依然有价值，适合探索与调试；批量流水线更适合用"固定输入→固定过程→固定输出"的方式运行。

### 学到的第二点：把 AI 当成初级工程师带，效果会明显提升

我观察到：很多 AI（包括表现不错的模型）并不会自动补齐你默认会做的工程步骤，比如写文档、写测试、做边界条件、更新变更记录。

如果你希望它按工程化方式产出，就需要把要求写清楚，像带一个初级工程师那样拆任务与验收标准。

对我有效的一套流程是：

1. 先让它给出 planning，尽量细到步骤与风险点
2. 我确认方向后再让它实现，避免它"先写再说"
3. 先设计 test cases，再写实现
4. 测试跑通后，把关键设计与用法写回项目文档，保证以后能捡起来

即使模型的"代码分数"未必最高，只要协作流程清晰，产出仍然可以达到非常高的可用度。

### 学到的第三点：让项目自解释，能显著降低"重新捡起来"的摩擦

我很喜欢 Claude 的 `/init` 那种"先读取项目上下文再开始工作"的体验，于是我也把类似习惯写进项目规范：

- 进入工作目录先看是否存在项目说明（例如 OpenCLAW.md）
- 每做完一个阶段，把变更与下一步更新回这个说明

现在项目还小，收益已经能感觉到；等项目变大，这种"持续维护上下文"的价值会更明显。

## 当天的一个小插曲

今天我还听了一期播客，嘉宾在讲他使用 OpenClaw 的过程里遇到的各种问题。他的描述非常具体：就像在带一个小孩写作业，一边生气，一边又被它的进步惊到。

我听的时候一直在笑，同时也在反思：当我们在软件里增加 AI 能力时，很多"不会写代码的人"遇到的卡点，其实来自流程与预期管理，而不是模型聪明程度。这个提醒非常实用。

## 结尾

即将进入 AI Native 时代, 让你的 AI 更了解你会变得重要。这个小项目不仅仅让我尝试了目前火热的 OpenClaw, 试用了很多以前没用过的工具, 让我更了解目前 AI 的进展, 也是将来让 AI 了解我的一小步。假期最后一天把这个心愿补上，我很满足。
